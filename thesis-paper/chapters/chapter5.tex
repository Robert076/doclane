\chapter{User workflow}
\label{chap:ch5}

\par This chapter describes how different types of users interact with the system and walks through the typical sequences of actions for the most common scenarios.

\section{Registration and login}

\par Before accessing any part of the system, a user must create an account by providing their name, email address, and a password. Once the account is created, the user can log in immediately. The session established after login determines which features are available, since the system enforces role-based access and different roles expose different functionality.

\section{Citizen workflow}

\par After logging in for the first time, the user is prompted to complete their profile with contact details such as address and phone number. This information is attached to requests the user submits, so the institution has the necessary context without requiring the user to re-enter it every time.

\par Once the profile is set up, the user can browse the available request templates and select the one that matches what they need. Each template specifies what information must be provided and which supporting documents need to be uploaded. The user submits the request, and then fills in all required fields.

\begin{center}
    \includegraphics[width=0.8\textwidth]{figures/chapter5/request-submission.png}
    \captionsetup{type=figure}
    \captionof{figure}{Submitting a request.}
\end{center}

\par After submission, the request is forwarded to the relevant department. The user can follow its progress from their dashboard, where the current status is visible at any point. If the reviewing employee rejects one or more documents, the user receives a notification along with the reason for each rejection, and can upload corrected versions directly through the same interface. The request then re-enters the review queue without requiring any further action from the user beyond uploading the corrected files.

\begin{center}
    \includegraphics[width=0.8\textwidth]{figures/chapter5/document-uploaded.png}
    \captionsetup{type=figure}
    \captionof{figure}{Request status as seen by the citizen.}
\end{center}

\par When all documents have been approved, the request is marked as completed and the user is notified. If the nature of the request requires collecting a physical document from the institution, this is the point at which the user would do so.

\section{Department employee workflow}

\par Requests submitted by citizens are routed to the department responsible for that type of request. All members of that department can see incoming requests, but before beginning a review an employee claims the request, which assigns it to them and signals to colleagues that it is already being handled. This avoids duplicate effort and keeps responsibility clear within the team.

\begin{center}
    \includegraphics[width=0.8\textwidth]{figures/chapter5/request-view-admin.png}
    \captionsetup{type=figure}
    \captionof{figure}{Department request inbox.}
\end{center}

\par Once a request is claimed, the employee reviews each uploaded document individually. Documents can be approved or rejected independently of one another. When rejecting a document, the employee is required to provide a reason, which is then surfaced to the citizen as feedback. This loop continues until every document in the request has been approved, at which point the request is closed.

\section{Administrator workflow}

\par Administrators manage the request templates that define what citizens can submit. Creating a template involves specifying the required document slots, the information fields the citizen must fill in, and optionally attaching example files for each slot to give citizens a reference for what an acceptable upload looks like. Templates can be tagged to group related types of requests together and make them easier to find both for citizens browsing the list and for administrators managing a growing catalogue.

\section{AI-assisted document processing}

\par The system integrates three AWS managed services to support document processing during the review stage. These are fully managed, serverless APIs and involve no custom model training or dataset curation on the part of this project. The backend makes API calls to each service at the appropriate point in the workflow and uses the response to assist the reviewing employee.

\par Textract is used for text extraction. When a document is uploaded as a scan or a photo, the backend sends it to Textract, which performs OCR (Optical Character Recognition) and returns the text content in a structured form. This allows the employee to read the document content without having to manually inspect an image file, and makes the extracted text available for further processing.

\par Bedrock provides access to large language models through a managed inference API. Doclane uses the Amazon Nova Lite model to assess whether an uploaded document actually corresponds to what the request template requires. The backend sends the text extracted by Textract alongside a description of the expected document, and the model returns an assessment. This gives the employee an additional signal when reviewing documents that are ambiguous or low quality, though the final decision always rests with the employee. The model is used as-is through the API, with no fine-tuning or modifications applied.

\par Polly converts text to synthesised speech. The extracted text from a document can be passed to Polly, which returns an audio file. This is primarily useful for accessibility purposes or for reviewing longer documents without reading through them in full.

\par None of these services make or record final decisions within the system. Their outputs are surfaced to the employee as contextual information, and every approval or rejection is performed explicitly by the assigned department member.
