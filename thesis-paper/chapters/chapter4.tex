\chapter{Deployment}
\label{chap:ch4}

\par Deploying an application involves a range of decisions that compound on each other: where the application runs, how it is packaged, how traffic reaches it, and how the infrastructure itself is managed. For Doclane, all of these decisions pointed toward a cloud-based setup, specifically on AWS, using containers orchestrated by Kubernetes and infrastructure defined entirely as code.

\section{Cloud computing and provider selection}

\par The case for cloud computing over self-hosted infrastructure is largely practical. Buying and maintaining physical servers introduces overhead that has nothing to do with building the application itself: procurement, setup, ongoing maintenance, and the risk of hardware failure with no built-in redundancy. Renting compute from a cloud provider sidesteps most of that. Resources can be provisioned when needed and released when they are not, and the billing follows actual usage rather than a fixed capital cost.

\par AWS was the chosen provider for two reasons. It holds the largest share of the global cloud market, roughly 31\% as of 2026, which reflects both the breadth of its service catalogue and the maturity of those services after years of iteration. The second reason is that the author holds an AWS Solutions Architect Professional certification and was already familiar with the platform, which reduced the ramp-up time considerably when making architectural decisions.

\section{The architecture}

\par The system runs inside an EKS (Elastic Kubernetes Service) cluster deployed within a VPC across two availability zones. Incoming traffic is handled by an ALB (Application Load Balancer) that routes requests either to the backend or the frontend depending on the path. The backend connects to an RDS PostgreSQL instance for persistent storage and an S3 bucket for uploaded documents. Authentication is delegated to Cognito.

\subsection{Containers and Kubernetes}

\par Both the backend and frontend are packaged as Docker containers using multi-stage builds. The backend Dockerfile compiles the Go source into a fully static binary in the first stage, then copies it into a minimal distroless base image that contains no shell, no package manager, and no unnecessary dependencies. The resulting image is approximately 36 MB and runs as a non-root user. The frontend follows the same two-stage pattern: the first stage runs the Next.js build, and the second stage copies only the standalone server output into a Node.js Alpine image, producing a final image of around 370 MB. The NEXT\_PUBLIC environment variables, things like the Cognito pool ID and backend URL, are passed as build arguments at build time since Next.js inlines them into the JavaScript bundle and they are not sensitive values.

\par Kubernetes manages the lifecycle of these containers across the cluster. Rather than manually starting containers on individual machines, the desired state is declared, for example two replicas of the backend, and Kubernetes reconciles reality to match that declaration, handling restarts, health checks, and rescheduling automatically. EKS offloads the management of the Kubernetes control plane itself to AWS, so the operational concern is limited to the worker nodes and the workloads running on them.

\par The backend runs as a Deployment with two replicas by default, each serving the HTTP router on port 8080. A HorizontalPodAutoscaler scales this between two and six replicas based on CPU utilization, adding capacity when average usage crosses 70\%. Liveness and readiness probes are configured so that crashed containers are restarted and traffic is only routed to pods that are ready to serve. The frontend is deployed similarly with two to four replicas, and server-side rendering requests from the frontend to the backend travel over the internal Kubernetes network using the service DNS name, never leaving the cluster.

\subsection{Networking}

\par The VPC is provisioned in the eu-west-1 region and spans two availability zones, with both private and public subnets in each. Worker nodes and the RDS instance sit in private subnets with no direct route to the public internet. The ALB and NAT Gateway are placed in the public subnets: the ALB receives inbound traffic from the internet, and the NAT Gateway allows resources in private subnets to reach the internet for things like pulling container images, without being reachable from it in return. The distinction between public and private subnets comes down entirely to the route table: a public subnet has a direct route to the internet gateway, while a private subnet routes outbound internet traffic through the NAT Gateway.

\par The subnets carry Kubernetes-specific tags so that the AWS Load Balancer Controller can automatically discover which ones to place ALBs in when an Ingress object is created inside the cluster.

\par Network security is enforced through Security Groups. The RDS instance is protected by a group that permits inbound connections on port 5432 only from the EKS cluster Security Group, meaning only pods running inside the cluster can reach the database. All traffic between the backend and RDS is encrypted over SSL/TLS, and traffic between pods stays within the internal Kubernetes network and never leaves the VPC.

\subsection{Ingress, DNS, and HTTPS}

\par The AWS Load Balancer Controller runs inside the cluster and watches for Ingress objects. When one is created, the controller provisions a real ALB in AWS configured to match the routing rules it defines. For Doclane, the Ingress routes requests to \texttt{/api} to the backend service and everything else to the frontend. The ALB terminates TLS using an ACM certificate, redirects plain HTTP to HTTPS, and routes traffic directly to pod IPs using IP-mode target groups rather than going through node ports.

\par The application is served at \texttt{thesis.robert-beres.com}. The ACM certificate for this domain was validated via DNS and attached to the ALB. A Route53 alias record points the domain to the ALB, and when the infrastructure is torn down and recreated, Terraform updates the record automatically to reflect the new ALB DNS name.

\subsection{Authentication}

\par User registration, login, password policy enforcement, and token issuance are all handled by Cognito. The frontend uses the Amplify library to interact with Cognito directly from the browser. After a successful login, Cognito issues a JWT that the frontend stores in an HTTP-only cookie, keeping it inaccessible to JavaScript and removing the most common vector for token theft via XSS. The backend validates incoming tokens by checking their signature against Cognito's published public keys.

\subsection{Document storage}

\par Uploaded documents are stored in S3 with versioning and server-side encryption enabled. The backend never exposes permanent credentials to the client; instead, it generates presigned URLs, which are time-limited links that allow a client to upload or download a specific object directly from S3. When the URL expires it cannot be reused, and the IAM permissions scoped to the backend role ensure that even a leaked presigned URL can only be used for the specific operation it was generated for.

\subsection{Backend credentials and least privilege}

\par The backend pods require access to several AWS services. Rather than embedding static credentials in the container image or environment variables, the application uses IRSA (IAM Roles for Service Accounts), which links a Kubernetes Service Account to an IAM Role. When a pod runs under that Service Account, the AWS SDK automatically receives short-lived credentials scoped to exactly the permissions the role defines. The attached policy follows the principle of least privilege: it permits uploading and downloading from the documents bucket, calling Textract for text extraction, calling Bedrock for document interpretation, calling Polly for text-to-speech synthesis, and looking up users in Cognito. Nothing beyond that is permitted.

\subsection{Database layer}

\par RDS was used for the database rather than running PostgreSQL directly on a cluster node. The managed service handles operating system patching and provides automated backups without any additional configuration. The database schema is applied on every fresh deployment by a Kubernetes Job that runs the initialization SQL script, and a separate Job bootstraps the admin user, so the entire environment can be torn down and recreated without manual intervention.

\subsection{Logging}

\par Container logs from all pods are collected by Kubernetes and accessible via kubectl. The backend emits structured JSON logs, which makes filtering for specific events or errors straightforward without needing a separate log aggregation service during development.

\section{AWS managed services for intelligent document processing}

\par Doclane integrates three AWS managed services to add document processing capabilities to the workflow. These are fully managed, serverless offerings: no model training was done, no datasets were curated, and there is no custom ML pipeline. The application makes API calls to these services and receives results back, with all the underlying infrastructure and model maintenance handled by AWS.

\par Textract is used to extract text from uploaded documents. When a document is submitted, the backend sends it to Textract and receives a structured representation of the text content, which can then be used for downstream processing or validation.

\par Polly converts text to synthesised speech. This is used to generate audio from document content, allowing users to listen to a document rather than read it, which can be useful for accessibility purposes or for quickly reviewing a lengthy submission.

\par Bedrock provides access to large language models through a managed API. Doclane uses the Amazon Nova Lite model to interpret whether a document uploaded by a user actually matches what the request template requires. The backend sends the extracted text alongside the expected document description and receives a structured assessment in return. The model is not fine-tuned or modified in any way; it is prompted at inference time with context specific to the request being validated.

\section{Infrastructure as Code}

\par Provisioning cloud infrastructure manually through the AWS console is intuitive for one-off tasks, but it does not scale well as a practice. Changes made by clicking through a UI leave no audit trail, are difficult to reproduce exactly, and become hard to reason about as the architecture grows. Infrastructure as Code addresses this by treating infrastructure the same way source code is treated: declared in files, version controlled, and applied through a repeatable process.

\par Terraform was chosen for this. Unlike CloudFormation, which is AWS-specific, Terraform is provider-agnostic and has a large ecosystem of providers. It was also chosen for practical ergonomics: the ability to manage AWS resources, Kubernetes manifests, and Helm charts from a single tool with a consistent workflow was more convenient than stitching together multiple tools.

\subsection{State management}

\par Terraform tracks what it manages in a state file that maps declared resources to their real counterparts in AWS. This state is stored remotely in an S3 bucket with versioning enabled so it can be recovered if corrupted. A DynamoDB table serves as a lock to prevent concurrent modifications from two different operators or CI runs from leaving the state inconsistent.

\subsection{Stack layout}

\par The infrastructure is split into four stacks, each with its own state file. The \textbf{bootstrap} stack creates the S3 bucket and DynamoDB table used for state management itself and is run once and never destroyed. The \textbf{platform-data} stack contains long-lived, inexpensive resources: ECR container registries, the Cognito user pool, the ACM certificate, and the GitHub OIDC role used for CI. These persist across compute teardowns so that users do not lose their accounts and the certificate does not need to be revalidated every time the rest of the infrastructure is recreated. The \textbf{platform-compute} stack contains the expensive resources: VPC, NAT Gateway, EKS cluster, RDS database, S3 documents bucket, and IRSA roles. The \textbf{workload} stack contains everything deployed inside the cluster: Helm charts, Kubernetes manifests, the database seeding job, and the admin bootstrap job. The compute and workload stacks are destroyed together when the application is not in use.

\subsection{Ephemeral environments}

\par The compute layer is designed to be torn down and recreated freely. Running \texttt{make up} provisions the full environment from scratch in roughly 15 minutes, including the VPC, EKS cluster, RDS instance, container deployments, database schema, admin user, ALB, HTTPS termination, and DNS record. Running \texttt{make down} destroys everything in around 10 minutes, reducing the running cost to zero. This is practical during development when leaving a full environment running overnight serves no purpose.

\section{Continuous Deployment}

\par The deployment pipeline is fully automated using GitHub Actions. When changes are pushed to the main branch, the pipeline builds Docker images for the backend and frontend independently, only rebuilding a service when its source code has actually changed. Images are tagged with the git commit SHA for traceability and pushed to ECR.

\par The pipeline authenticates to AWS using GitHub's OIDC integration rather than storing static credentials anywhere. The GitHub Actions runner generates a signed identity token, which AWS validates against a configured OIDC provider in IAM. If the token checks out and originates from the Doclane repository, the runner is permitted to assume a role whose only permission is pushing images to the two ECR repositories. The credentials generated through this role assumption are short-lived and expire after the job completes, so even if they were somehow extracted from a build log they would be useless shortly after.
